# 旧阅读索引框架审计

## 结论

新方向是“续读恢复与当前理解”，旧框架中最大的问题是把产品默认重心放在“阅读索引”上，尤其是人物、地点、时间线、关系这些实体型入口。这会让任何书籍都被诱导成实体抽取问题，而不是阅读恢复问题。

## 高优先级问题

### 1. 默认入口仍是阅读索引

阅读页左侧有“阅读索引”入口，展开后优先出现人物、地点、时间线、组织、关系等面板。这和新需求冲突，因为续读恢复不应从索引开始，而应从恢复情境开始。

处理策略：隐藏默认阅读索引入口。底层索引能力保留，未来作为“按需图形组织器”或证据检索能力使用。

### 2. AI 阅读文案仍强调人物、地点、时间线

AI 面板和分析摘要仍写着“提取人物、地点、时间线与关系”。这会强化错误产品心智。

处理策略：改成“续读恢复”，强调关键点、前置理解、主动回忆和证据。

### 3. 分析完成弹窗仍按人物/地点/关键事件展示

分析摘要弹窗按人物、组织、地点、关键事件分组，不适合科普、技术、商业、哲学等类型。

处理策略：第一阶段停用自动弹出旧分析摘要。后续改为续读恢复卡。

### 4. 段落辅助仍有“前文/出处”的旧抽屉

当前段落辅助 drawer 是“本地阅读提示 + ContextCite + 相邻原文 + 出处”，可以保留一部分，但默认说明“实体与日期从本地识别”不符合新方向。

处理策略：后续改造成“当前页前置理解”。第一阶段先不作为默认入口扩张。

## 中优先级问题

### 5. 关系图作为显式功能过早

关系图对历史、传记、小说有价值，但不应是默认入口。它应由内容结构触发。

处理策略：保留代码，隐藏默认入口，只在恢复卡或类型分析判断需要时出现。

### 6. 自动分析模式语义不清

旧自动分析是“每读 N 页更新索引”，新方向应该是“后台准备续读恢复材料”。

处理策略：改名和文案，后续再调整底层任务。

## 可复用资产

- 已读边界记录。
- 阅读位置恢复。
- RAG evidence store 与 ContextCite。
- 书籍类型识别。
- 关系图和时间线组件，但只能按需出现。
- 书签和笔记数据，但需要转化为记忆锚点或主动回忆问题。

## 第一阶段清理范围

1. 隐藏阅读索引默认入口。
2. AI 阅读入口改为“续读恢复”。
3. AI 面板文案改为恢复卡导向。
4. 不再自动弹出旧的人物/地点/事件分析摘要。
5. 新增续读恢复卡 MVP 入口，作为读者重新打开书后的第一体验。
